# System Audit Report — TournamentHQ

## Critical Issues (Bugs)

### 1. Umpire email still visible in sidebar for umpire
- **File:** `includes/header.php:120-121`
- The email was removed for all roles, but the umpire user avatar link points to `admin/index.php` instead of the umpire profile/dashboard. Umpires don't have a profile page, so clicking the avatar sends them to admin.
- **Fix:** For umpires, the avatar link should go to `/TournamentHQ/umpire/dashboard`.

### 2. Play page tournament cards don't link anywhere
- **File:** `play.php:327-344`
- The tournament cards show "View →" but are plain `<div>` elements, not `<a>` tags. Clicking does nothing.
- **Fix:** Either make cards link to tournament bracket or a detail page.

### 3. Play page stats are hardcoded
- **File:** `play.php:296-315`
- Stats show "500+ Players", "50+ Tournaments", "10K Matches" — these are static, not pulled from the database. Misleading to users.
- **Fix:** Query actual counts from DB.

### 4. Umpire nav links missing from header
- **File:** `includes/header.php:60-66`
- The `$navLinks` array has entries for admin, organizer, and player roles, but NO entry for umpire. When an umpire is logged in, the sidebar shows no navigation links at all.
- **Fix:** Add umpire nav links (Dashboard at minimum).

### 5. Organizer recent results shows guest winner names as empty
- **File:** `organizer/index.php:123-125`
- The recent results query uses `winner_first`/`winner_last` from the match SQL, but when a guest wins, those fields may be empty because the SQL join doesn't always resolve guest names.
- **Fix:** Verify the `matchSelectSql()` JOIN handles guest winners properly.

### 6. Player join flow doesn't register the account player
- **File:** `player/index.php:70-76`
- The registration modal says "Your account profile is not included in the tournament" and only registers guests. The player's own account is never added to the tournament_players table. This means the logged-in player never plays — they can only register other people.
- **Fix:** Either auto-register the account player or clarify the UX.

### 7. Admin manage_matches.php missing from nav
- **File:** `includes/header.php:42-49`
- Admin nav has Users, Players, Tournaments, Auto Bracket, Reports — but no link to `admin/manage_matches.php` (the file exists).

---

## Missing Features / Gaps

### 8. No 404 error page
- No custom 404 page exists. Invalid URLs show a default server error page.

### 9. No player notifications for admin
- **File:** `includes/header.php:11-14`
- Notification badge only works for organizer and player roles. Admin and umpire roles have no notification support.

### 10. No match scheduling UI
- Matches are auto-created by bracket generation but there's no UI to set `match_date` or `table_number` individually. The only way to assign these is via direct DB editing.

### 11. No tournament completion flow
- When all matches in a tournament are completed, there's no action to mark the tournament as "completed". It stays "ongoing" forever.

### 12. No walkover/forfeit button
- The match status ENUM includes 'walkover' but there's no UI button to mark a match as walkover. Only "Record Result" (completed) exists.

### 13. No CSRF token protection
- POST forms don't use CSRF tokens. An attacker could craft a form on another site that submits match results to the umpire endpoint.

### 14. Players.points column never updated
- **File:** `modules/matches/match_functions.php`
- `adjustPlayerStatsForMatchResult()` updates `wins` and `losses` but never touches the `points` column, making it permanently 0 for all players.

### 15. No email verification resend on login
- **File:** `includes/auth.php:126-133`
- When an unverified user tries to login, they get a generic "email_not_verified" error but there's no UI to resend the verification email from the login page.

---

## Security Concerns

### 16. Umpire can access any tournament
- The umpire dashboard allows selecting ANY ongoing tournament, not just the one they're assigned to. An umpire for Tournament A could score matches in Tournament B.
- **Severity:** Medium — depends on trust model.

### 17. SQL injection in bracket_view.php
- **File:** `includes/bracket_view.php:13`
- `db()->query("SELECT is_team_event FROM tournaments WHERE id = " . (int)$tid)` — while the `(int)` cast prevents injection, it's not using prepared statements.

### 18. No rate limiting on login
- No protection against brute force login attempts.

### 19. Payment proof files not access-controlled
- **File:** `modules/uploads/payment_proof.php`
- Payment proof files in `uploads/payment_proofs/` may be directly accessible via URL without authentication.

---

## UI/UX Issues

### 20. Landing page doesn't show login CTA
- **File:** `index.php:506-536`
- The landing page has "Play" and "Organize" nav buttons but no direct login button or CTA for existing users.

### 21. Sidebar doesn't collapse on mobile properly
- **File:** `includes/header.php:133-135`
- The sidebar toggle button exists but needs JS to actually toggle the sidebar class. If the JS is missing or broken, mobile users can't navigate.

### 22. Flash messages don't auto-dismiss
- **File:** `includes/header.php:152-158`
- Flash messages have a manual close button but no auto-dismiss timer. Users may miss success messages.

### 23. Umpire dashboard has no navigation sidebar
- The umpire dashboard doesn't include a sidebar navigation — it's a standalone page with no way to navigate to other sections or back to login.

### 24. Inconsistent page titles
- Header shows "TT Tournament Manager" for all pages. Should show the actual `$pageTitle`.

---

## Code Quality

### 25. Massive inline styles
- Nearly every page uses extensive inline `style=""` attributes instead of CSS classes. This makes maintenance difficult and increases page size.

### 26. Duplicated landing page CSS
- `index.php`, `play.php`, and `organize.php` each define their own `:root` variables and layout CSS inline (~200+ lines each). These should share a common public stylesheet.

### 27. No input sanitization on guest registration
- **File:** `player/index.php:52-67`
- Guest first/last names are passed directly to `addTournamentGuest()` without sanitization beyond `trim()`. XSS risk if names are displayed without `e()`.

### 28. Error handling in match recording
- **File:** `modules/matches/match_functions.php:336`
- `recordBracketMatchResult()` calls `getTournamentById()` which can fail silently. If the tournament doesn't exist, the code continues without the double-elimination check.
